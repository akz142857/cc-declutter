# cc-declutter v0.2 — Data-Driven Recommendations

**Status:** Ready for implementation
**Target release:** 0.2.0 (next minor after 0.1.0)
**Owner:** ziy
**Design decisions log:** see [Resolved decisions](#resolved-decisions) at bottom

## Problem

v0.1.0 whitelist is built from a user-declared stack string (e.g. `ios,go,ts,python`).
This has two failure modes observed in real runs:

1. **Over-deletion.** Users forget to declare a stack they actually use.
   A run with stack `ios,go,ts,python,java,rust` still proposed deleting
   `web`, `kotlin`, and `frontend-slides` — all of which the user had invoked
   within the past 30 days.
2. **Under-deletion.** Declared stacks keep skills the user has never actually
   triggered (e.g. `kotlin-ktor-patterns` kept because `android` was declared,
   but the user has not touched Kotlin in 6 months).

v0.2 replaces the "declared-stack-only" heuristic with a **usage-informed**
recommendation engine backed by local telemetry the CLI already writes.

## Non-goals

- **Not** a full analytics system. No network calls, no external storage.
- **Not** a replacement for user judgement. We still emit a reviewable
  `prune.sh` with `DRY_RUN=1` default.
- **Not** automatic. The user still runs `plan` explicitly.
- **Not** a plugin manager. v0.2 reports plugin utilization but will not
  generate plugin-uninstall commands (see [Q4 resolution](#resolved-decisions)).

## Data sources

All local, all already on disk:

| Source | Path | Signal |
|---|---|---|
| Prompt history | `~/.claude/history.jsonl` | User prompts with timestamps + project. Shows slash-command invocations (`/foo`), skill name mentions in prose. |
| History earliest timestamp | `~/.claude/history.jsonl` first line | Floor on how far back usage signals can reach. Drives new-user degradation (see bands). |
| Bash log | `~/.claude/bash-commands.log` | Every `Bash` tool call. Shows skill-triggered shell commands by path pattern. |
| Session transcripts | `~/.claude/projects/*/*.jsonl` | Ground truth for `Skill` tool invocations. Scanned via ripgrep pre-filter — cheap when accelerated. |
| Settings | `~/.claude/settings.json` | Hook commands that reference skills (e.g. `observe.sh` → `continuous-learning-v2`). |
| Plugin registry | `~/.claude/plugins/installed_plugins.json` | Which skills live inside active plugins vs. orphaned. |

History files are append-only JSONL/plain-text. Parsing cost is O(file size),
one-shot per `audit` or `plan` run. On the sample machine:
`history.jsonl` ≈ 576 KB, `bash-commands.log` ≈ 691 KB, transcripts ≈ 711 MB.
With ripgrep pre-filter, transcript scan is ~10 s (not the 60–120 s naive estimate).

## Usage detection algorithm

For each skill `S` in `~/.claude/skills/` (and active plugin caches):

### Signals (ordered strongest to weakest)

1. **Hook dependency.**
   Parse all hook `command` strings in `settings.json`.
   If the command path contains `skills/<S>/` or runs a script that
   references `<S>`, mark `<S>` as hook-pinned.
   Weight: **pin** — never recommend removal regardless of other signals.

2. **Transcript `Skill` tool call.** *(default-on, ripgrep-accelerated)*
   Pipeline:
   - `rg -l '"name":"Skill"' ~/.claude/projects/ -tjsonl` → list of files with any Skill invocation
   - Filter list by mtime within window
   - For each surviving file, `rg -n '"name":"Skill"'` to get line numbers
   - Parse only matched lines as JSON, extract `input.skill` + timestamp
   - Fallback chain: ripgrep → grep → skip-with-warning
   Weight: **strong** — direct evidence, catches auto-invocations that slash-command signal misses.

3. **Slash-command invocation.**
   Regex `/\b/(?P<cmd>[\w-]+)\b/` against `history.jsonl[].display`.
   Slash name → skill name via each plugin's manifest `commands` entries.
   Handles plugin-namespaced commands (`everything-claude-code:prp-pr` → `prp-pr`).
   Weight: **strong** — explicit user intent.

4. **Bash command pattern.**
   Skill SKILL.md files sometimes declare `allowed-tools` or reference
   characteristic scripts. Match `bash-commands.log` lines against skill
   script paths.
   Weight: **medium**.

5. **Natural-language mention.**
   Skill name appears as bare token in `history.jsonl[].display`.
   Weight: **weak** — ambiguous. Used only as a tiebreaker, never as sole evidence.

### Confidence bands

After collecting signals, each skill lands in one of four bands:

| Band | Criteria | Default action |
|---|---|---|
| `pinned` | Hook-referenced, or self (`cc-declutter`) | Keep, never prompt |
| `active` | Any strong signal within window | Keep |
| `stale` | No strong signal in window, but in declared stack whitelist | Keep, commented out in prune.sh |
| `dormant` | No strong signal in window, not in whitelist | Propose delete |

`stale` surfaces "you declared you need this but never use it" so users can
consciously drop it rather than silently keeping it.

### Window strategy

**Default window: 90 days.** Rationale:
- Matches quarterly work cadence (most common sprint rhythm)
- 30 days was too aggressive — one vacation + context switch miscategorizes
- User-overridable via `--window-days N`

**Multi-window report.** Audit always displays 7 / 30 / 90 day counts side by side:

```
Skill                     7d   30d   90d   Band
swiftui-patterns          12   47    142   active
kotlin-ktor-patterns      0    0     3     stale      ← borderline
laravel-patterns          0    0     0     dormant    ← safe to drop
```

This exposes the gradient so users can override the band when they know
context the tool doesn't (e.g. "I'm starting a Kotlin project next week").

**New-user degradation.** If `history.jsonl` earliest entry < 50% of the
window (default: < 45 days for 90-day window), audit emits a warning and
`plan` refuses to produce dormant deletions:

```
⚠ Only 14 days of history available (< 45d = 50% of window).
Usage signals unreliable. Falling back to stack-only mode.
Re-run in a few weeks for usage-informed recommendations.
```

The `Banding` result carries a `degraded: bool` flag that propagates into
the plan script, suppressing the entire dormant section when true.

### Plugin utilization

|--------------------------|--------|--------|-------------|-------------|

## Plan changes

Plan output gains four things:

1. **Confidence-banded sections.** The generated `prune.sh` is now grouped
   by band, not one flat list:

   ```bash
   # === Dormant skills (0 usage in 90 days, not in stack) ===
   # Safe high-confidence removals. Review and pass.
   run 'rm -rf ~/.claude/skills/article-writing'    # last seen: never (slash:0, transcript:0, bash:0)
   ...

   # === Stale skills (in stack, 0 usage in 90 days) ===
   # You declared these but haven't triggered them. Commented out by default.
   # Uncomment to remove.
   # run 'rm -rf ~/.claude/skills/kotlin-ktor-patterns'  # stale: last seen 2025-10-14 (slash only)
   ...

   # === Orphaned plugin registry entries ===
   # Safe: entries point to non-existent cache dirs. Claude Code ignores these anyway.
   run "python3 -c \"...remove orphan entries from installed_plugins.json...\""

   # === Protected (hook-referenced, skill engine) — DO NOT DELETE ===
   # Informational only. Listed so you can audit the exclusion logic.
   # continuous-learning-v2  (PreToolUse, PostToolUse)
   # cc-declutter            (self)
   ```

2. **Per-skill usage annotation.** Every `rm` line gets a trailing comment
   with last-seen date and the strongest signal type.

3. **Hook-dependency section in the report.** Before generating the plan,
   print any skill saved because of a hook reference.

4. **Degraded mode.** When new-user degradation triggers, dormant section
   is omitted entirely; script contains only `rules/zh`, orphan plugin
   entries, and SessionStart hook proposal (all usage-independent).

## Architecture

Add two new pure modules alongside v0.1 code.

```
lib/
├── detect.py       # existing — filesystem inventory
├── whitelist.py    # existing — stack → skill mapping
├── usage.py        # NEW — parse history/log/transcripts → per-skill signals
├── bands.py        # NEW — combine signals + whitelist → bands
└── plan.py         # existing — render prune.sh (now consumes bands)
```

### Constraints on the new modules

- **Python stdlib only.** No pip dependencies. Preserves cc-declutter's
  "clone and run" story.
- **Pure functions + frozen dataclasses.** No shell calls, no filesystem
  writes inside `usage.py` / `bands.py`. Callers do IO.
- **Ripgrep invocation** lives in a thin adapter (`lib/rg.py` or inline in
  usage.py) with documented fallback to `grep`.
- **Testable in isolation** with fixture JSONL blobs — see Test plan.

### Key types

```python
@dataclass(frozen=True)
class Signal:
    kind: Literal["hook", "transcript", "slash", "bash", "mention"]
    timestamp: date
    detail: str          # e.g. "slash=/swiftui-patterns" or "transcript:line=347"

@dataclass(frozen=True)
class UsageRecord:
    skill: str
    last_seen: date | None
    signals: tuple[Signal, ...]    # ordered strongest-first
    hook_pinned: bool
    count_7d: int
    count_30d: int
    count_90d: int

@dataclass(frozen=True)
class Banding:
    pinned: frozenset[str]
    active: frozenset[str]
    stale: frozenset[str]
    dormant: frozenset[str]
    degraded: bool                 # True when history is too short
    degraded_reason: str | None
```

### Key functions

```python
# usage.py
def scan(claude_home: Path, window_days: int = 90,
         deep_scan: bool = True) -> dict[str, UsageRecord]: ...

def earliest_history_entry(history_path: Path) -> date | None: ...

# bands.py
def categorize(usage: dict[str, UsageRecord],
               whitelist: set[str],
               history_earliest: date | None,
               window_days: int) -> Banding: ...
```

## CLI surface

No new subcommands. v0.1 modes all consume the new data.

**New flags on `plan`:**
- `--window-days N` (default `90`) — history lookback window
- `--no-usage` — skip usage scan, fall back to v0.1 stack-only behavior
- `--no-deep-scan` — skip transcript scan (falls back to history/bash/hooks only)
- `--include-stale` — move `stale` skills into the active removal block
  instead of commented-out

**New flags on `audit`:**
- `--window-days N`
- `--no-deep-scan`

## Safety rails

Unchanged from v0.1, plus three new rules:

1. **Backup directory and `RESTORE.md` unchanged.**
2. **`DRY_RUN=1` default unchanged.**
3. **A skill can only be proposed for deletion if it is `dormant`**, OR the
   user passed `--include-stale`. No other code path produces `rm` lines
   for a skill.
4. **Hook-pinned skills cannot be deleted regardless of flags.** If a user
   really wants to delete one, they must first remove the hook (and the
   script tells them exactly which hook to remove).
5. **Plugin uninstalls never generate `rm` lines.** v0.2 reports utilization
   but only removes *orphaned registry entries* (entries pointing to
   non-existent cache dirs). Live plugins are untouched.
6. **Degraded mode suppresses dormant section entirely.** No usage-based
   deletion when history is too short to be trusted.

## Test plan

### Fixtures

- `tests/fixtures/history-heavy.jsonl` — 500 entries, 20 distinct slash commands, spans 120 days
- `tests/fixtures/history-sparse.jsonl` — 10 entries, no slash commands, 10 days span
- `tests/fixtures/history-new-user.jsonl` — 3 entries, 5 days span (triggers degraded mode)
- `tests/fixtures/bash-log-mixed.log`
- `tests/fixtures/transcripts/` — 3 synthetic project dirs with mock Skill tool calls
- `tests/fixtures/settings-with-hooks.json`
- `tests/fixtures/installed_plugins-with-orphans.json`
- `tests/fixtures/skills/` — 10 mock SKILL.md files

### Unit tests

- `usage.scan()` returns expected bands for each fixture combination
- Hook-pinned detection survives both `matcher: "*"` and explicit-matcher hooks
- Slash-command → skill resolution handles plugin-namespaced commands
- Window-days boundary: event at day 29 counted, event at day 31 not counted
  (test for both 30-day and 90-day windows)
- Multi-window counts: same fixture scanned at 7/30/90d produces expected counts
- Ripgrep fallback: mock `rg` missing → falls to `grep` → falls to skip-with-warning
- New-user degradation: `earliest_history_entry < 50% of window` sets
  `Banding.degraded=True`
- Orphan plugin detection: entry in `installed_plugins.json` with missing
  cache dir is flagged

### Integration tests

- End-to-end on a synthetic `~/.claude/` tree → assert generated `prune.sh`
  has no `rm` lines outside the `dormant` section (unless `--include-stale`)
- Degraded mode: new-user fixture → `prune.sh` has no dormant section
- Snapshot test: audit output is stable across runs when input is unchanged
- Ripgrep-off path: `--no-deep-scan` produces a report with a clearly-marked
  "transcript scan skipped" note

## Rollout

1. Land `lib/usage.py` + `lib/bands.py` + tests. Skill still uses v0.1 logic.
2. Wire `audit` to call new modules. Transcript scan on by default;
   `--no-deep-scan` exists as escape hatch from day one.
3. Wire `plan` to emit banded `prune.sh`.
4. Dogfood on the author's own `~/.claude/` (currently 165 skills, 711 MB
   transcripts — ideal stress test).
5. Dogfood with 2–3 external users.
6. Release 0.2.0.

## Out of scope for v0.2

- **Interactive `plan --interactive`** (v0.3)
- **Windows support** (v0.4)
- **Cross-machine usage aggregation** — not planned
- **Plugin uninstall recommendations** — deferred to future RFC.
  v0.2 reports plugin utilization but does not generate removal commands
  for live plugins (only for *orphaned registry entries*, which is zero-risk).

## Resolved decisions

This section records the four open questions that were resolved before
implementation started. Kept for future reference when revisiting trade-offs.

### Q1: `lib/` Python modules vs. inline heredocs — **Resolved: split `lib/`**

Rationale:
- v0.3 interactive mode requires state (confirmed set, pending queue,
  audit log) — heredocs cannot carry this cleanly.
- Usage detection correctness depends on edge cases (window boundaries,
  plugin-namespaced slash parsing, hook command path matching) that need
  unit tests to stay correct across refactors.
- v0.1's inline heredocs are already at readability ceiling.

Constraints locked in:
- Python stdlib only; no pip dependencies.
- Pure functions + frozen dataclasses; no IO in core modules.
- `install.sh` copies `lib/` alongside `SKILL.md`.

### Q2: Window default — **Resolved: 90 days + multi-window report + new-user degradation**

Rationale:
- 30 days misjudges users on quarterly cadences; one vacation breaks it.
- Adaptive windows sacrifice explainability and behave badly on sparse users.
- 90 days matches common sprint rhythms; multi-window report (7/30/90)
  surfaces borderline cases so users can override.
- New-user mode (history < 50% of window) degrades gracefully instead of
  producing wrong recommendations from insufficient data.

### Q3: Transcript scanning — **Resolved: default-on with ripgrep**

Rationale:
- Original design assumed 60–120 s scan cost. Benchmarked alternative
  (ripgrep pre-filter + targeted JSON parse) runs in ~10 s on 711 MB.
- Transcript is the only signal that catches auto-invoked Skill tool calls,
  estimated at ~50 % of real skill usage. Skipping it misses the main signal.
- Graceful fallback chain (`rg` → `grep` → skip-with-warning) handles
  missing binaries without failing the whole run.
- `--no-deep-scan` exists as opt-out for users with unusually slow disks
  or privacy concerns about transcript reading.

### Q4: Plugin-level uninstall recommendations — **Resolved: report only, defer action to future RFC**

Rationale:
- Plugin scope mismatch: v0.2 operates on skills; plugin uninstall is a
  different blast radius (commands, agents, hooks all affected).
- Uninstall cost often exceeds benefit — losing 147 skills to reclaim 5 k
  tokens when 12 are active is a net loss.
- 90-day usage data is insufficient to judge plugin lifetime value
  (upcoming project needs, etc.).
- `/plugin` UI already provides the official uninstall path.

v0.2 adds orphaned-plugin-entry cleanup as a zero-risk bonus (registry
entries pointing to missing cache dirs) — this is a clear bug, not a
judgement call.
